# QuantKing 协作约束

本文件是本仓库后续人工开发、Codex 会话和其他自动化代理的强制工作约束。开始任何开发前，先阅读本文件及任务相关文档；若与旧代码冲突，以 V3 文档为准，除非任务明确要求维护旧行为。

## 权威文档顺序

1. `docs/DECISIONS.md`：已确认的架构决策与产品边界。
2. `docs/V3_IMPLEMENTATION_PLAN.md`：阶段顺序、范围和验收条件。
3. `docs/TECHNICAL_STANDARDS.md`：代码、数据、测试和 Web 技术规范。
4. `docs/DATA_ARCHITECTURE.md`、`docs/RESEARCH_WORKFLOW.md`：领域规范。
5. `docs/ARCHITECTURE.md`：模块边界和迁移规则。
6. `docs/CHANGELOG.md`、`docs/quant-packages-api-reference.md`：历史参考，不得覆盖前述决策。

## 产品边界

- 单用户、本机、A 股日频中长线研究与调仓决策工具。
- 数据源为可替换适配器；XTQuant 仅为迁移期来源，AKShare 为免费来源之一。
- 输出调仓计划，用户手动通过券商下单并登记成交。
- 不实现自动下单、实时交易、Tick/盘口、高频、多用户、云端服务或微服务。
- Web 仅监听 `127.0.0.1`；不为不在范围内的认证、CORS 或远程 API 设计复杂方案。

## 不可破坏的规则

1. 业务层、研究层和回测层不得直接导入 `xtquant`、`akshare` 或 FastAPI `Request`。
2. 原始行情不可覆盖；派生数据、实验、回测和调仓计划必须记录其输入版本。
3. 数据源差异必须显式记录，禁止静默使用备用源覆盖主源。
4. 因子、标签和信号只能使用信号日已知的数据；任何时间泄漏视为阻断问题。
5. 模型只生成分数，组合层才生成目标权重和调仓计划；信号不得直接转为券商订单。
6. 新代码不向旧的自动交易主链路（`trading/`、`StrategyExecutor`）增加功能。
7. 没有对应测试和文档更新的架构变更不得合入。

## 开发流程

1. 明确任务属于 V3 的哪个阶段；不要跳过前置验收条件。
2. 先写或更新接口/数据契约及测试，再实现适配器或业务逻辑。
3. 保持改动小且可回滚；不重写无关模块，不删除历史数据。
4. 运行与改动相关的测试；跨模块改动运行 `pytest tests -q -p no:cacheprovider`。
5. 更新相应文档：接口变更更新技术规范，范围/决策变化更新 `DECISIONS.md`，阶段完成更新路线图。

## 文件操作与提交前检查

- 使用 `apply_patch` 编辑和删除项目文件。
- 不执行 `git reset --hard`、`git checkout --` 或删除用户未授权的数据文件。
- 保留无关的工作树改动；当前仓库的本地数据库和行情数据不应纳入 Git。
- 最低检查：`git diff --check`、相关 pytest、Markdown 链接检查。
